RMAN Quick Reference

Backup & recovery commands you'll reach for every week.
Phase 4 — Handle the Fires · Kudrant Lab

What this is: The essential RMAN commands for daily backup operations and common recovery scenarios. This covers the 80% of RMAN you'll use 95% of the time. Keep this bookmarked — it's your safety net reference card.

When to use it: When you need to run a backup, check backup status, or recover from a failure. For full deep-dive coverage of every RMAN scenario, see the Ultimate RMAN Guide in The Shop.

1

Connect to RMAN

Start here every time
Command line
# Connect to RMAN as the target database rman target / # Connect with a catalog database (if you use one) rman target / catalog rman_user/password@catalog_db # Connect to a remote target rman target sys/password@target_db
2

Run a Full Database Backup

Your most important command
When: Schedule this nightly. Also run it manually before any major change (patching, upgrades, structural changes). This is your insurance policy.
RMAN — Full backup with archive logs
-- Full database backup + archive logs + autobackup of controlfile BACKUP DATABASE PLUS ARCHIVELOG; -- Full backup with a tag for easy identification BACKUP DATABASE TAG 'FULL_DAILY_20260622' PLUS ARCHIVELOG; -- Full backup compressed (saves disk space) BACKUP AS COMPRESSED BACKUPSET DATABASE TAG 'FULL_COMPRESSED' PLUS ARCHIVELOG;
✅ Do
  • Always include PLUS ARCHIVELOG — you need the logs for recovery
  • Use meaningful tags with dates so you can find backups later
  • Use compression if disk space is a concern
🚫 Never
  • Skip the archive logs — without them, point-in-time recovery is impossible
  • Run full backups during peak hours without approval
3

Run an Incremental Backup

Faster daily backups
When: Use incremental backups between full backups to save time and space. Level 0 = base (like a full). Level 1 = only blocks that changed since the last Level 0 or Level 1.
RMAN — Incremental backup strategy
-- Level 0: the baseline (run weekly, e.g., Sunday) BACKUP INCREMENTAL LEVEL 0 DATABASE TAG 'INCR_L0_SUNDAY' PLUS ARCHIVELOG; -- Level 1 Differential: changed blocks since last L0 or L1 (run daily) BACKUP INCREMENTAL LEVEL 1 DATABASE TAG 'INCR_L1_DAILY' PLUS ARCHIVELOG; -- Level 1 Cumulative: ALL changed blocks since last L0 (alternative) BACKUP INCREMENTAL LEVEL 1 CUMULATIVE DATABASE TAG 'INCR_L1_CUM' PLUS ARCHIVELOG;
Common strategy

Sunday: Level 0 (full baseline). Monday-Saturday: Level 1 differential (only changes). This gives you fast daily backups with full recovery capability.

4

Check Backup Status

Part of your morning check
RMAN — View backup history
-- List all backups from the last 7 days LIST BACKUP COMPLETED AFTER "SYSDATE-7"; -- Summary of backup history LIST BACKUP SUMMARY; -- Check if database is recoverable REPORT NEED BACKUP; -- Show any backups that are expired or obsolete REPORT OBSOLETE;
SQL — Alternative: query from SQL*Plus
-- Quick backup status from SQL (same as the morning check query) SELECT session_key, input_type, status, TO_CHAR(start_time, 'YYYY-MM-DD HH24:MI') AS start_time, TO_CHAR(end_time, 'YYYY-MM-DD HH24:MI') AS end_time FROM v$rman_backup_job_details WHERE start_time > SYSDATE - 7 ORDER BY start_time DESC;
5

Validate a Backup

Trust but verify
Why validate: A backup file can exist on disk but be corrupt. Validation reads every block and confirms the backup is actually usable for recovery. Run this periodically — a backup you can't restore from is not a backup.
RMAN — Validate backups
-- Validate the most recent backup RESTORE DATABASE VALIDATE; -- Validate a specific backup by tag RESTORE DATABASE FROM TAG 'FULL_DAILY_20260622' VALIDATE; -- Crosscheck: verify backup files still exist on disk CROSSCHECK BACKUP; CROSSCHECK ARCHIVELOG ALL; -- Delete expired (files no longer on disk) DELETE EXPIRED BACKUP; DELETE EXPIRED ARCHIVELOG ALL;
6

Restore & Recover (Common Scenarios)

When things go wrong
Important: These are the most common recovery scenarios. For complex recovery (point-in-time, tablespace, block-level), see the Ultimate RMAN Guide in The Shop. Always have a plan before you start recovering.
Scenario 1 — Restore a single datafile
-- Datafile is missing or corrupt SQL> ALTER DATABASE DATAFILE '/path/to/datafile.dbf' OFFLINE; RMAN> RESTORE DATAFILE '/path/to/datafile.dbf'; RMAN> RECOVER DATAFILE '/path/to/datafile.dbf'; SQL> ALTER DATABASE DATAFILE '/path/to/datafile.dbf' ONLINE;
Scenario 2 — Full database restore and recovery
-- Complete restore (database must be mounted, not open) RMAN> STARTUP MOUNT; RMAN> RESTORE DATABASE; RMAN> RECOVER DATABASE; RMAN> ALTER DATABASE OPEN;
Scenario 3 — Restore the controlfile
-- Controlfile is lost or corrupt RMAN> STARTUP NOMOUNT; RMAN> RESTORE CONTROLFILE FROM AUTOBACKUP; RMAN> ALTER DATABASE MOUNT; RMAN> RECOVER DATABASE; RMAN> ALTER DATABASE OPEN RESETLOGS;
✅ Do
  • Have a plan before you start — know what you're restoring and why
  • Take a fresh backup immediately after a successful recovery
  • Document every step of the recovery for future reference
🚫 Never
  • Start a recovery in production without understanding the scenario
  • Skip RECOVER after RESTORE — restore puts files back, recover applies logs to make them current
  • Forget to back up again after OPEN RESETLOGS — the log sequence reset means old backups won't chain forward
7

Manage Backup Retention & Cleanup

Keep your backup area healthy
RMAN — Retention and cleanup
-- Set retention policy: keep backups for 14 days CONFIGURE RETENTION POLICY TO RECOVERY WINDOW OF 14 DAYS; -- Show current RMAN configuration SHOW ALL; -- Delete obsolete backups (outside retention window) REPORT OBSOLETE; DELETE OBSOLETE; -- Delete archive logs already backed up (free space) DELETE ARCHIVELOG ALL COMPLETED BEFORE 'SYSDATE-7';
Good housekeeping

Run CROSSCHECK and DELETE OBSOLETE weekly to keep your backup area from growing indefinitely. Set a retention policy that matches your recovery requirements — 7 days minimum, 14-30 days recommended.

The RMAN golden rule: A backup you haven't validated is a hope, not a backup. Run RESTORE VALIDATE periodically — it's the only way to know for sure your backup actually works. The worst time to discover a corrupt backup is when you need it.
